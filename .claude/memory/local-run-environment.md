Running `examples/embodiment/train_embodied_agent.py` (e.g. `run_embodiment.sh kitchen4_composition_vmpo_smoke`) locally on this box (single RTX 4090, 24 GB):

- **Venv:** the training stack lives in `./openpi` (uv venv, python 3.11). `source openpi/bin/activate` first. The `libero-max` conda env is sim-only (no hydra/openpi). LIBERO is on PYTHONPATH via the venv activate.
- **Checkpoint:** `models/Pi05-LIBERO-SFT` (downloaded from HF `RLinf/RLinf-Pi05-LIBERO-SFT`, 7 GB). Smoke config points at `${REPO_PATH}/models/Pi05-LIBERO-SFT`.
- **EGL render fix (required):** MuJoCo env workers fail with "Cannot initialize a EGL device display / failed to create dri2 screen" unless you force the NVIDIA EGL vendor:
  `export __EGL_VENDOR_LIBRARY_FILENAMES=/usr/share/glvnd/egl_vendor.d/10_nvidia.json` (plus `MUJOCO_GL=egl`, `MUJOCO_EGL_DEVICE_ID=0`). This replaces the apt/EGL setup in `requirements/embodied/sys_deps.sh`, which is skipped here (no sudo).
- **ptrace blocker (the real wall):** collocated single-GPU runs hardcode CUDA-IPC weight sync (actorâ†’rollout same device), which calls `pidfd_getfd` and needs `kernel.yama.ptrace_scope=0`. This machine has it set to `1` â†’ `RuntimeError: pidfd_getfd: Operation not permitted` during the first weight sync. Fix needs root: `sudo sysctl -w kernel.yama.ptrace_scope=0` (persist via `/etc/sysctl.d/`). No code/config toggle to force NCCL instead. Until set, the pipeline runs end-to-end up to the first weight sync then dies here.
  - **The IPC-vs-NCCL switch is `_check_same_device_with_peer()` in `rlinf/scheduler/collective/collective_group.py`:** returns `1` (â†’ CUDA-IPC, needs ptrace) only when actor and rollout EACH see exactly one GPU (collocated single-GPU); returns `-1` (â†’ NCCL, no ptrace) when they're on different GPUs. So **disaggregated 2-GPU placement (actor GPU0, rollout GPU1) sidesteps the ptrace blocker entirely** and works even in unprivileged containers.
  - **vast.ai unprivileged CONTAINER instances cannot run collocated:** `ptrace_scope` is read-only (sysctl change blocked **even with sudo** â€” `sudo sysctl -w kernel.yama.ptrace_scope=0` â†’ "ignoring: Read-only file system"), no `CAP_SYS_PTRACE` (verified: sibling `pidfd_getfd` â†’ "Operation not permitted" even as root). So a single-GPU vast *container* is blocked regardless of GPU memory (tested on A100-80GB containers, both 1-GPU and 2-GPU). Only a vast **VM** instance (kernel access) allows the collocated IPC path; in a container you MUST use disaggregated 2-GPU placement.
  - **CONFIRMED WORKING (2026-06-22): disaggregated 2-GPU run completes end-to-end on 2Ă— A100-SXM4-40GB (NVLink, 256 GB disk, Ubuntu 24.04, driver 580/CUDA 13).** Config `kitchen4_composition_vmpo_2gpu.yaml` (actor GPU0 ~21 GB, rollout GPU1 ~8.7 GB) ran `Global Step 2/2` clean: `vmpo/eta` 1.0â†’1.02, `top_half_fracâ‰ˆ0.52`, `returns_mean` 0.317â†’0.84, `critic/value_loss` 0.0086â†’0.0031, `explained_variance` -0.22â†’0.016. Launch env: `MUJOCO_GL=egl`, `__EGL_VENDOR_LIBRARY_FILENAMES=/usr/share/glvnd/egl_vendor.d/10_nvidia.json`, `NCCL_P2P_DISABLE=1`, then `bash examples/embodiment/run_embodiment.sh kitchen4_composition_vmpo_2gpu`.
  - **The earlier "illegal memory access" was the 2Ă— A100-80GB PCIe box (topology `NODE`, no NVLink). Root cause: `NCCL_P2P_DISABLE=1` wasn't reaching Ray workers** â€” fixed by the `nvidia_gpu.py` patch (commit de3bf5c) that forwards NCCL_* env vars to each worker. With NVLink (this SXM4 box) P2P is reliable anyway. **Disk must be â‰¥100 GB**: the 32 GB box completed Step 1/2 then died on `save_file: No space left on device` (Ray spill / checkpoint), NOT a code bug.
  - **Hardware guidance:** memory-wise the actor peaks ~21 GB, so any 2Ă— Ampere/Hopper â‰¥24-40 GB works (A100/A6000/L40S/A40). Avoid RTX 5090 (Blackwell sm_120) unless you upgrade torchâ†’2.7+/cu128 + rebuild flash-attn â€” the pinned torch 2.6.0+cu124 lacks Blackwell kernels.
  - **Five fixes were needed to get the multi-GPU run working (all in commit de3bf5c on V-MPO):** disaggregated placement (ptrace), `nvidia_gpu.py` NCCL var propagation (illegal-memory), `action_utils.py` `libero_ltl_composition` dispatch (NotImplementedError), `io_struct.py` reach/cost-rewards omit-when-empty (None concat crash), `dual.py` eta device alignment (cuda/cpu mismatch in the V-MPO temperature dual).
- **24 GB is NOT enough for collocated runs (confirmed over many attempts).** Even with every memory lever â€” `rollout.enable_offload`, `actor.enable_offload`, `actor.fsdp_config.cpu_offload`, `gradient_checkpointing`, `best_of_n: 1`, `micro/global_batch_size: 1`, `total_num_envs: 1`, `camera 128`, `mixed_precision.param_dtype: bf16`, `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` â€” it OOMs at the **first actorâ†’rollout CUDA-IPC weight sync**, consistently ~1 GB over (actor ~15.5 GB fp32 master + rollout ~5.9 GB + a ~1 GB IPC clone buffer; ~890 MB free). bf16 FSDP `param_dtype` does NOT shrink the peak because FSDP keeps an fp32 master copy. The IPC sync requires the actor's full weights resident on GPU (can't offload during the send), so the peak is unavoidable on one card. Everything else runs: install, config, Ray/placement, model load, EGL render, env/sampler/AP-prompt/reward, V-MPO optimizer init, and the IPC sync itself (after the ptrace fix). **Need ~32 GB+ (e.g. 5090-32GB, A100/L40S/A6000 40-48 GB) to clear the weight-sync peak**, or a disaggregated 2-GPU placement (actor on GPU0, rollout on GPU1) which avoids same-device IPC entirely. Valid precision tokens: `bf16`/`fp16`/`fp32` (NOT `bfloat16`); see `rlinf/config.py:torch_dtype_from_precision`.

Install was done via `requirements/install.sh embodied --venv openpi --model openpi --env libero` (note: `--env libero`, not the README's stale `maniskill_libero`). The sudo `sys_deps.sh` step was skipped (EGL configs already present); LIBERO editable + openpi git + prebuilt flash-attn wheel + transformers_replace patch were run manually after.
